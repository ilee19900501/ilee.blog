mq概念

多用于分布式系统之间进行通信。
基础架构：

broker：mq服务器，分发和接受消息的应用
virtual host：虚拟主机，出于多租户安全考虑，当多个用户使用同一个server时可以创建多个virtual host来进行隔离（类似于数据库里面的database）
connection：生产者和消费者和broker之间的tcp链接
channel：如果每一次访问mq都建立conn,消耗很大，channel是mq内部建立的逻辑连接，如果应用支持多线程，每个thread会创建单独的channel进行通讯
exchange：message到达broker的第一站，根据分发规则，匹配routingkey到对应queue，常见类型：fanout(广播)，direct（1对1）,topic（通配符）
Binding:队列与交换机之间的虚拟连接。

优势：解耦，异步提速，削峰填谷
异步提速：比如下订单服务，对应着减库存，支付。同步操作的中间加了一层mq，下订单之后把订单信息发送到mq，库存和支付服务从里面取消息。
削峰填谷：比如秒杀场景，同一时间有4000个请求，我的订单服务只支持1000并发量，此时就可以先把4000个请求放到mq里面去，建立多个消费者每秒从mq里面拉去1000个请求去处理。
缺点：系统可用性降低，复杂性提高。
要解决的问题？1.如何保证mq高可用，宕机了（搭建集群）2.如何保证消息不丢失（持久化，消息补偿机制）

mq工作模式

1.workQueue 工作队列 （一个生产者，一个队列，多个消费者会轮询消息，每个消费者获取到的消息唯一）

2.发布订阅模式 生产者需声明一个fanout的交换机，会广播到所有队列中去

应用场景：up主发布博客之后，所有粉丝都能收到

3.topic通配符模式

消息可靠性投递

1.生产者的confirm，return模式
return消息机制：如果发送的消息到达不了目标队列(当前交换机不存在或路由key到达不了队列，或网络原因)，会触发这种监听。
可把这种消息重发或者存到数据库人工处理
使用：实现ReturnCallback,配置文件设置spring.rabbit.publish.returns=true
confirm消息机制：消息成功或失败发送到exchange（设置错误的交换机）都会触发这种监听。
使用：实现ConfirmCallback接口，配置文件设置spring.rabbit.publish.confirm-type=corrlated
可把这种消息重发或者存到数据库人工处理。
confirm机制下是拿不到消息内容的，只是起到一个通知作用。所以即使消息失败生产者在这回调函数里也无法重发。可以借助redis去实现这种的消息重发。
思路：第一种：1.生产者发送消息的时候，生成唯一ackId,
2.redis里面根据ackId，存储消息内容，并设置过期时间
3.redis过期回调，重复1，2
4.消息成功则删除消息
第二种：消息发送失败后，入库统一人工处理（以为这种消息失败很少）

消费者ack

1。怎么保证消息的幂等性，由于网络原因，生产者向队列多次发送同一条数据
解决：第一种1.可以使用redis的setnx
第二种2.乐观锁机制，比如发送的消息是对数据库的更新操作，可以加一个version版本号，更新一次版本号+1
2.消费者处理消息失败 （如果消费时候，代码出现异常导致消息丢失）
解决：加入重试机制，失败之后重新入队，3次尝试失败之后，直接拒绝，进入死信队列，再统一处理
概念：1> 消息称为死信的条件？如果消息ack失败没有到达死信队列则会丢失
1.队列拒绝channel.basicReject或者channel.basicNack重新入队为false
2.ttl（消息存活时间）到期
3.队列消息数量超过最大长度

如何用rabbitmq实现延迟队列

方法：死信队列+ttl（消息存活时间）
场景：互联网下订单后如果30min未支付，则取消订单
解决档案：1.使用定时器，每隔30min去轮询数据库。缺点：一个订单下单后29min，然后定时任务刚好执行，那么得等到下次轮询时才会取消这个订单。每隔1min刷则系统消耗太大
2.使用延迟队列可完美解决，下单后，发到延迟队列里面30min后去处理消息

如何处理消息积压问题

1.消息积压是如何产生的
网络原因或者消费者能力不足
解决：采用工作队列模式，上线更多的消费者，将消息批量取出来放到数据库在做处理

rabbitmq安装

环境阿里云centos8，特别注意elang与rabbitmq版本兼容问题

  1. #安装erlang
    wget https://github.com/rabbitmq/erlang-rpm/releases/download/v22.0.4/erlang-22.0.4-1.el7.x86_64.rpm
    yum -y install erlang-22.0.4-1.el7.x86_64.rpm
  2. 下载rabbitmqwget https://github.com/rabbitmq/rabbitmq-server/releases/download/v3.8.1/rabbitmq-server-3.8.1-1.el8.noarch.rpm
    yum -y install rabbitmq-server-3.8.1-1.el8.noarch.rpm
  3. 开启管理界面 rabbitmq-plugins enable rabbitmq_management
  4. 修改配置文件使guest可以远程访问
    /usr/lib/rabbitmq/lib/rabbitmq_server-3.8.1/ebin/rabbitmq.app
    将39行的：{loopback_users, [<<”guest”>>]} ，改为：{loopback_users, []}
  5. /usr/lib/rabbitmq/bin 目录下 ./rabbitmq-server start
  6. 浏览器访问即可 http://ip:15672/ 账号和密码都是guest
    确定erlang和rabbitmq 正确安装后无法远程连接web管理界面。
