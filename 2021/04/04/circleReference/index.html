spring循环依赖分析

Posted on 2021-04-04 Edited on 2021-04-19 In spring5源码
desc:spring循环依赖

为什么会有循环依赖

解决循环依赖（手写循环依赖解决）

1.使用一级缓存
Map<String, Object> singtonObjects = new ConcurrentHashMap<>();

分析：上面代码使用一级缓存在bean实例化之后就放到一级缓存里面，每次getBean(),先去一级缓存里面去拿，解决了循环依赖问题。但是在多线程场景下，会拿到不完整的bean（这时候instance2的实例是空的），因为是在bean实例化之后就放到一级缓存里面去的。

2.使用二级缓存
// 一级缓存主要存放成熟的bean
static Map<String, BeanDefinition> beanDefinitionMap = new HashMap<>();
// 二级缓存主要存放纯净的bean（实例化后的）
static Map<String, Object> earlySingtonObjects = new ConcurrentHashMap<>();

分析：使用二级缓存解决了循环依赖，但是还有一个问题，如果Instance1实现了aop动态代理，此时从二级缓存里拿到的bean只是Instance1的实例，这样就会有问题，所以引入了三级缓存（二级缓存也可以解决aop问题，但是代码存在耦合，违反spring单一设计原则，因为需要在循环依赖的逻辑（上面第二个判断）里去创建动态代理，然后返回动态代理的bean）
说明：正常bean，spring在初始化之后会创建动态代理，循环依赖的需要在bean实例化之后创建动态代理

3.使用三级缓存
// 一级缓存主要存放成熟的bean
static Map<String, BeanDefinition> beanDefinitionMap = new HashMap<>();
// 二级缓存主要存放纯净的bean（实例化后的）
static Map<String, Object> earlySingtonObjects = new ConcurrentHashMap<>();
// 三级缓存主要存放动态代理的bean（如果没有动态代理，返回传进来的）
static Map<String, ObjectFactory> objectFactory = new ConcurrentHashMap<>();

分析：使用三级缓存，存放函数接口ObjectFactory,获取动态代理的bean，完全解耦。
–动态代理获取bean代码分析
1.SmartInstantiationAwareBeanPostProcessor bean的后置处理器，获取bean之前通过代理获取bean的实例。spring动态代理也实现了这个接口
2.如果有动态代理则走代理，否则返回传进来的bean

拓展：如果在另一个方法中，不需要方法传参对某个对象去判断（减少代码耦合），这里可以引入函数接口实现

问题：
1.构造方法的循环依赖可以解决吗？
使用构造方法时候，bean都没有实例化完成，所以没办法解决
2.propoType下可以解决循环依赖吗？
propoType下bean都没有放到缓存中去
3.使用三级缓存之后还有可能获取到不完整的bean

解决方案：加锁
